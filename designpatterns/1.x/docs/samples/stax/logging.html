6000 - Log/Mon - Logging Standards & Practices

6000 - Log/Mon - Logging Standards & Practices - DESCRIPTION

Logging provides business auditing, diagnostic and development information, operational data, and security auditing information for NextGen applications. Logging information is broken down into three main categories – application logging, application support infrastructure logging, and system logging. There are a variety of tools being used within USPTO to collect, aggregate, and analyze the information from all of these logging mechanisms. Proper logging process must be followed in order to allow these tools to operate.

6000 - Log/Mon - Logging Standards & Practices - USAGE

Application designers must:

  1. Ensure that the application employs components that can be properly instrumented to work with the USPTO standard monitoring and auditing tools. Typically, these would include AppDynamics and Splunk.
  2. Ensure that the events specified can be logged through an appropriate mechanism.
  3. Some of these events may be logged by application servers, AppDynamics, etc. These will be referred to as “automatically” logged.
  4. Ensure that the application maintains a complete history of all completed business transactions. Applications that save complete transaction history may use the history mechanism for auditing purposes. Applications that don’t should log all business transactions.

Application developers must:

  1. Use the logging framework approved within their application technology stack.
  2. Log all events specified that the application designer has determined are not automatically logged.
  3. Check to ensure that the logging format adheres to the format specified

6010 - Log/Mon - Logging Framework

6010 - Log/Mon - Logging Framework - DESCRIPTION

Logging provides business auditing, diagnostic and development information, operational data, and security auditing information for NextGen applications. Logging information is broken down into three main categories – application logging(this category), application support infrastructure logging, and system logging. There are a variety of tools being used within USPTO to collect, aggregate, and analyze the information from all of these logging mechanisms. This category defines the application logging tools.

A logging framework is a computer data logging package for an application or service platform. Logging refers to the recording of activity. Logging is a common issue for development teams. Frameworks ease and standardize the process of logging for the development platform.

Logging is broken into three major component:

  • The Logger - Responsible for capturing the message to be logged along with certain metadata and passing it to the logging framework
  • Formatter - Formats logged message for output
  • Handler/Appender - Appends or outputs the formatted message to the appropriate end point.This might include a console display, writing to disk, appending to a database, or email.

6010 - Log/Mon - Logging Framework - USAGE

Application designers must:

  1. Instrument - Ensure the application employs components that can be properly instrumented to work with the USPTO standard monitoring and auditing tools (AppDynamics and Splunk)
  2. Rationalize - Ensure that the events can be logged through an appropriate mechanism
  3. Coverage - Ensure application maintains a complete history of all completed business transactions.

NOTE: Events logged by application servers, AppDynamics, etc. are referred to as “automatically” logged

NeGen applications use a standardized exception mechanism within their business logic. Built into the construction process for some standard exceptions should be the automatic generation of an appropriate log message. Thus, certain types of failures might throw a transaction failure exception, and the act of throwing that exception would automatically log the transaction failure (presumably with severity ERROR). The use of the common severity definitions and the enforcement of appropriate logging is a code level issue and should be addressed by code reviews. It will be extremely important that programmers follow this model if the Operations team is going to get useful information from the logs.

  • FATAL - Process has had an unrecoverable failure
  • ERROR - Process interrupted
  • WARNING - Indicates unusual occurence in completed process
  • INFO - Process or unit of work noteworthy in some manner
  • DEBUG - Information useful to developer or devops error management.

Types of events that must be logged

  • Transaction Failure
  • Critical Transactions
  • Calls to other systems
  • Calls to web services
  • Database calls
  • Important business transactions

6012 - Log/Mon - Logging Framework Patents Logback 1.1.x Logback 1.2.x

Logback is the successor to log4j, picking up where log4j leaves off. Logback's architecture is sufficiently generic so as to apply under different circumstances. At present time, logback is divided into three modules, logback-core, logback-classic and logback-access.

The logback-core module lays the groundwork for the other two modules. logback-classic natively implements the SLF4J API so that you can readily switch back and forth between logback and other logging frameworks such as log4j or java.util.logging (JUL). logback-access module integrates with Servlet containers, such as Tomcat and Jetty

6020 - Log/Mon - Application Performance Management

6020 - Log/Mon - Application Performance Management - DESCRIPTION

Application Performance Management (APM) is the monitoring and management of performance and availability of software applications. APM focuses on diagnosing complex application performance problems to maintain an expected level of service. APM is the translation of IT metrics into business meaning (value) Two sets of performance metrics are closely monitored. The first defines performance experienced by end users of the application. One example of performance is average response times under peak load. The components of the set include load and response time.
  • Load - The volume of transactions processed by the application, e.g., transactions per second (tps), requests per second, pages per second. Without being loaded by computer-based demands for searches, calculations, transmissions, etc., most applications are fast enough, which is why programmers may not catch performance problems during development
  • Response times - Times required for an application to respond to a user's actions at such a load

The Second metrics measure computational resources used by the application under load, indicating whether there is adequate capacity to support the load, as well as possible locations of a performance bottleneck. Measurement of these quantities establishes an empirical performance baseline for the application. The baseline can then be used to detect changes in performance. Changes in performance can be correlated with external events and subsequently used to predict future changes in application performance.

Gartner's APM Conceptual Model:

  • End user experience (EUE) – Measuring the transit of traffic from user request to data and back again is part of capturing the end-user-experience (EUE)
  • Runtime application architecture (ADDM) - Application Discovery and Dependency Mapping solutions exist to automate the process of mapping transactions and applications to underlying infrastructure components
  • Business transaction - Focus on user-defined transactions or the URL page definitions that have some meaning to the business community
  • Deep Dive Component Monitoring (DDCM) - Requires an agent install and is generally targeted in the middleware space focusing on the web, application, and messaging servers
  • Analytics/reporting - A common set of metrics to collect and report on for each application, then standardize on a common view on how to present the application performance data

6020 - Log/Mon - Application Performance Management - USAGE

The use of APM is common for Web & Service applications, which lend themselves best to the more detailed monitoring techniques. In addition to measuring response time for a user, response times for components of a Web application or Service can also be monitored to help pinpoint causes of delay. There also exist HTTP appliances that can decode transaction-specific response times at the Web server layer of the application.

6030 - Log/Mon - IT Operations Analytics (ITOA)

6030 - Log/Mon - IT Operations Analytics (ITOA) - DESCRIPTION

IT operations analytics (ITOA) is an approach or method to retrieve, analyze, and report data for IT operations. ITOA may apply big data analytics to large datasets to produce business insights. Also known as IT data analytics are primarily used to discover complex patterns in high volumes of often "noisy" IT system availability and performance data.

Gartner's five types of analytics technologies:

  • Log analysis
  • Unstructured text indexing, search and inference (UTISI)
  • Topological analysis (TA)
  • Multidimensional database search and analysis (MDSA)
  • Complex operations event processing (COEP)
  • Statistical pattern discovery and recognition (SPDR)

6040 - Log/Mon - Synthetic Monitoring - CATEGORY

6040 - Log/Mon - Synthetic Monitoring

6040 - Log/Mon - Synthetic Monitoring - DESCRIPTION

Synthetic monitoring (also known as active monitoring or proactive monitoring) is website/service monitoring performed using Web browser emulation or scripted recordings of Web transactions. Behavioral scripts (or paths) are created to simulate an action or path that a customer or end-user would take on a site. Those paths are then continuously monitored at specified intervals for performance, such as: functionality, availability, and response time measures.

Synthetic monitoring enables a webmaster or an IT/Operations professional to identify problems and determine if a website or Web application is slow or experiencing downtime before that problem affects actual end-users or customers. This type of monitoring does not require actual Web traffic so it enables companies to test Web applications 24x7, or test new applications prior to a live customer-facing launch. This is a good complement when used with passive monitoring to help provide visibility on application health during off peak hours when transaction volume is low. Combined with APM tools, synthetic monitoring provides deeper visibility into end-to-end performance, regardless of where applications are running.

6040 - Log/Mon - Synthetic Monitoring - USAGE

Because synthetic monitoring is a simulation of typical user behavior or navigation through a website, it is often best used to monitor commonly trafficked paths and critical business processes. Synthetic tests must be scripted in advance, so it is not feasible to measure performance for every permutation of a navigational path an end-user might take. This is more suited for passive monitoring.

Synthetic testing is useful for measuring uptime, availability and response time of critical pages and transaction (how a site performs from all geographies) but doesn't monitor or capture actual end-user interactions, see Website monitoring. This is also known as Active monitoring that consists of synthetic probes and Web robots to help report on system availability and predefined business transactions.
